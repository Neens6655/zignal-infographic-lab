# PRD: ZGNAL Infographic Lab — Growth Engine (pSEO + AEO)

> Generated by /prd on 2026-04-02
> Revised after Plato Council audit (2026-04-02) — added Phase 0 MVP, kill conditions, data validation, AEO metrics, foundation fixes

## Problem & Context

ZGNAL Infographic Lab is live at zgnal.ai with a world-class AI infographic generation pipeline (20 layouts × 20 styles = 400 combinations, 22 research sources, 7-stage pipeline). But the site has only **11 indexed pages** and zero content marketing infrastructure. Google has almost nothing to crawl, AI assistants don't know the product exists, and the 400 style/layout combinations — each a potential landing page — are invisible to search engines. This project adds two growth layers that turn existing product assets into a scalable traffic engine at near-zero cost.

## Users

**Primary user:** Marketing professionals, consultants, educators, and content creators searching for infographic creation tools — reached via Google, AI assistants (ChatGPT, Perplexity, Claude), and directory listings.

**Secondary user:** AI agents and LLM systems that recommend tools to their users — reached via structured data, llms.txt, and AEO-optimized content.

## Core Features (v1)

### Layer 1 — Programmatic SEO (pSEO)

1. **Style landing pages (20)** — `/styles/[slug]` pages for each visual style with showcase, description, use-cases, and CTA
2. **Layout landing pages (20)** — `/layouts/[slug]` pages for each layout type with showcase, description, best-for scenarios, and CTA
3. **Industry vertical pages (50+)** — `/create/[industry]-infographic` pages targeting "infographic for [industry]" keywords
4. **Use-case pages (30+)** — `/create/[use-case]-infographic` pages targeting "how to create [use-case] infographic" keywords
5. **Combo pages (top 20 live, rest static)** — `/templates/[style]-[layout]` pages showcasing specific style+layout combinations
6. **FAQ hub page** — `/faq` with structured FAQ schema covering top 30+ questions about infographic creation
7. **Blog/content system** — `/blog` with MDX support for long-form SEO content, tutorials, and comparison articles
8. **Dynamic sitemap** — Auto-generates sitemap entries for all programmatic pages (400+)

### Layer 2 — Answer Engine Optimization (AEO)

9. **llms.txt** — Machine-readable index at `/llms.txt` describing site capabilities for AI crawlers
10. **AI crawler whitelisting** — robots.txt updated to explicitly allow GPTBot, ClaudeBot, PerplexityBot, GoogleOther, Amazonbot
11. **FAQ Schema (JSON-LD)** — `FAQPage` structured data on FAQ hub and embedded in relevant landing pages
12. **Organization + WebSite schema** — Additional JSON-LD types beyond current SoftwareApplication
13. **40-word answer blocks** — Every landing page starts with a concise, extractable answer paragraph optimized for AI citation
14. **Internal linking mesh** — Every programmatic page cross-links to related styles, layouts, and use-cases
15. **IndexNow integration** — Instant ping to Bing/Yandex when new pages are published or updated

## User Flow (Organic Discovery)

1. User searches "marketing infographic maker" on Google or asks ChatGPT "best tool for creating infographics"
2. Google serves `/create/marketing-infographic` landing page; ChatGPT cites zgnal.ai from llms.txt + FAQ schema
3. User lands on a page with: hero answer block, style/layout showcase previews, feature highlights, social proof
4. User clicks CTA → redirected to `/chat` or main generator with pre-filled context
5. User generates their first infographic in 60 seconds
6. User bookmarks / shares → organic backlinks + return visits

## Non-Goals (v1 Scope Firewall)

Explicitly NOT building in v1:
- Competitor comparison pages (`/alternatives/canva`, `/vs/piktochart`) — deferred to v2
- Auto-social posting system (Postiz integration) — separate project
- Directory submission automation — manual process using Startup-Launch-List CSV
- User-generated content / community features
- Paid advertising or ad tracking pixels
- Multi-language / i18n support
- CMS or admin panel for content management (data files + MDX are sufficient)
- Analytics dashboard beyond Vercel Analytics (no Matomo/SerpBear yet)
- A/B testing infrastructure
- Email capture beyond existing waitlist form

## Implementation Phases

### Phase 0: Foundation Fix + MVP Validation (PARALLEL TRACKS)

**Objective:** Fix critical foundation bugs AND validate the pSEO approach with 40 pages before scaling to 400+. Two parallel tracks run simultaneously.

**Track A — Foundation Fixes (from Plato audit: 11 RED findings):**
1. Triage all 11 RED findings from 2026-03-07 audit by severity: security > accessibility > architecture
2. Fix security REDs: CSP `unsafe-inline` (R3), rate limiter silent fail (R11), `dataUrlToBlob` crash (R10)
3. Fix accessibility REDs: `prefers-reduced-motion` on all animations (CS1), touch targets (R4), contrast (R5), focus trap (R6)
4. Fix UX REDs: Pricing dead-end (R7), hero headline clarity (R8)
5. Add social proof: Gather 3-5 testimonials or trust signals, add to homepage + landing page template
6. Add `error.tsx` to each new route segment (/styles, /layouts, /create, /templates)

**Track B — MVP pSEO (40 pages only):**
1. Create `src/data/styles.json` — 20 entries extracted from reference files (slug, name, description, category, bestFor, keywords, relatedLayouts)
2. Create `src/data/layouts.json` — 20 entries extracted from reference files (slug, name, description, category, bestFor, keywords, relatedStyles)
3. Create Zod schemas for both data files (`src/data/schemas.ts`) — validate at build time
4. Create `/styles/[slug]/page.tsx` + `generateStaticParams()` + `generateMetadata()` — 20 style landing pages with ≥200 words unique content each
5. Create `/layouts/[slug]/page.tsx` + `generateStaticParams()` + `generateMetadata()` — 20 layout landing pages with ≥200 words unique content each
6. Update `src/app/sitemap.ts` to include 40 new URLs dynamically
7. Create `public/llms.txt` — Markdown index of site capabilities
8. Update `src/app/robots.ts` — whitelist AI crawlers (GPTBot, ClaudeBot, PerplexityBot, GoogleOther, Amazonbot)
9. Deploy and submit sitemap to Google Search Console + Bing Webmaster Tools

**Checkpoint:** 40 style/layout pages live, all return 200 with unique content, sitemap has 51 URLs, Lighthouse SEO ≥95 on 3 sample pages, ≥8 of 11 RED bugs resolved.

**KILL CONDITION:** If Phase 0 pages do not achieve top-50 ranking on ANY target keyword after 60 days → pause Phase 1, run 2-week backlink sprint (30-50 free directories from Startup-Launch-List) before proceeding.

**DO NOT CHANGE after this phase:**
- Data file Zod schemas (shape of JSON objects)
- URL patterns (`/styles/[slug]`, `/layouts/[slug]`)
- llms.txt format
- robots.txt structure

### Phase 2: Scale to 400+ Programmatic Pages

**Objective:** Expand from 40 validated pages to 400+ using industry/use-case/combo data, with unique content, internal links, and answer blocks.

**Depends on:** Phase 0 (40 pages validated, indexing confirmed) + Phase 1 (expanded data)

**Requirements:**
1. Create `/create/[slug]/page.tsx` + `generateStaticParams()` + `generateMetadata()` — 80+ industry/use-case pages with: 40-word answer block, recommended style+layout combos, example prompts, social proof section, CTA
2. Create `/templates/[slug]/page.tsx` — Top 20 style-layout combo pages with static preview images (NO Gemini API calls at build time — use pre-generated or placeholder images). Remaining combos use text-only showcase.
3. Create `/faq/page.tsx` — FAQ hub with all 30+ questions, `FAQPage` JSON-LD via next-seo, category filtering
4. Each page: unique `<title>`, `<meta description>`, canonical URL via `generateMetadata()`
5. Each page: ≥200 words of unique content (not shared template boilerplate)
6. Each page: ≥3 internal links to related content (built from explicit `relatedStyles`/`relatedLayouts` data relationships — no auto-linking)
7. Each page: 40-word answer block in first `<p>` tag for AI extraction
8. Update sitemap to dynamically include all programmatic pages
9. OG images: Use static template with text overlay via query params (`/api/og?title=X`). No build-time generation for 400+ images.
10. Add `staticGenerationTimeout: 120` to next.config.ts as safety net
11. Measure build time after adding all pages — if >30 min, switch to ISR for /create and /templates routes

**Checkpoint:** `next build` completes in <30min, sitemap shows 400+ URLs, any programmatic page returns 200 with ≥200 words unique content, Lighthouse SEO ≥95.

**DO NOT CHANGE after this phase:**
- Data file schemas from Phase 0/1
- URL patterns (slugs are permanent — changing them breaks SEO)
- Page template structure
